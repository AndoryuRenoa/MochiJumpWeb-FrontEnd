The Web App (Non-Technical)

This web app is over-enginered to show case the capabilities of a modern web app

The first thing you may notice is that this app has a loading screen. It is possible to hide this and have this type of app look like it's loading like a normal webpage, but it's doing some very cool things when someone visits it for the first time.

This is an example of a Single Page App (SPA). Despite the page having different URLs, it is all actually layers on top of layers of the same page. This means that after the initial loading is complete the app will not have to load anything else until you leave the web app or unless it has to ask the server for information not initially loaded in the app package (think security or communicating with another server or saving something). The other nice thing about a design like that is everything is modular, so if you want to change one thing that should always be visible you only have to change it in one instance

The old prototype for the level editor is an example of a page built outside of the SPA


The Server (Non-Technical)

The server is what allows the website to actually do things. A well designed server is critical for a well functioning web experience as well as for any app that wants to talk to anything more than just itself

The server has many roles. One of the most common roles is that it can store, save, load, and deliver information. It can also perform business logic, which depending on your business could be anything from sending out automated emails to compiling any kind of concievable report on demand. Creating rules for accessing content is a critical aspect of security that is controlled by the server.

I love writing back end code. For the end user it's something that they never see, but always feel. If a web page does anything, it is because the server had code written to allow that functionality. In the case of this website the server has specially written code that allows it to:

  • Have specific user access
  • Have security
  • Save, load, and deliver levels for the Mochi Jump game
  • Send out emails when required
  • And deliver the Web-App in a single package the way it does.

    • There are many different options for writing server code. I, personally, use Java, cloud computing, and microservices. This gives me the ability to write apps that can scale with demand, and to seperate different functions into different environments to keep things manageable and to allow for rapid development if neccessary.


    Security (Non-Technical)

    There are four major aspects to security:

  • Having the user connect to the webpage securely
  • Having restrictions on what the user can access
  • Control what is sent and when
  • Protecting yourself from potential vulnerabilities

    • The first aspect of having the user connect to your site securely is extremely critical. This is the "s" in "https://" and the lock next to the URL in your browser when your on a secure webpage. If those components are missing then anyone can view all the infromation you are sending to the server as well as all the information the server is sending you. So, if you access a webpage that is not encrypted and you give it a username and password, guess what, you just freely shouted out loud that username and password for anyone who cared to listen

    The second aspect is what people normally think of when they think security. The server has to set the rules with regard to who can access what. It's a critical component to having security if you need it.

    The third aspect is where people can get themselves into trouble if they don't think about what they are doing. Under the web app section, we talked about a cool feature of modern design where you can send a package that contains the entire website (all pages included) when a user first visits the page. The downside to that is nothing in that package is secured. All the data and information is given freely. You can design an app that hides this from a browser, but it is still in that package and can be easily found by anyone who knows what they are doing. This is why it is critical that security is kept in mind in all aspects of development.

    The last aspect is keeping your stuff safe from hackers. To put it simply, if you take certain shortcuts when designing a server you could open up the potential for someone to inject code where it doesn't belong.
